# Security Architecture — JarBees (summary)

Date: 2026-07-14

Scope
- Threats addressed: Prompt injection, tool/agent injection, RAG poisoning, SQL/vector injection, privilege escalation, malicious uploads, data exfiltration.

Principles
- Zero-trust ingestion: Treat all external content as untrusted until curated and scored.
- Least privilege: Tools and agent actions run with minimal permissions; destructive actions require explicit human approval.
- Immutable audit trail: All actions that change state are logged with context, user/session id, and LLM prompts used.
- Deny-by-default for dynamic code/SQL execution: LLMs may output structured intents only; no raw SQL or shell commands allowed.

Mitigations

1. LLM Layer
- Output schema enforcement: LLM responses that map to actions must conform to a JSON schema and be validated before execution. Use a strict JSON Schema (intent, action, params) with types and allowed values.
- Prompt hardening: Strip user-supplied "instruction-like" fragments from documents before including them in prompts (regex blacklist for "ignore previous", "delete database", "execute", "run shell").
- Model policy: Prefer read-only or offline models for summarization; reserve models that can generate code for sandboxed environments only.
- Rate-limits and amplification guards: limit how many agent actions a single session can trigger in a time window.

2. Tool / Agent Execution
- ActionExecutionGate: central gate that enforces:
  - Role-based authorization (RBAC)
  - Human-in-the-loop (HITL) for destructive actions (delete, drop, export, grant)
  - Whitelist of allowed tool names and allowed parameters per tool
  - Parameter validation and sanitization
- Sandbox: run external code or tool invocations in an isolated environment with network and filesystem restrictions.
- Audit logs: every action request, approval, and outcome is written to `logs/security_audit.log`.

3. RAG / Ingest
- Source reputation and quarantine: assign a trust score per source; newly ingested public content goes to quarantine until reviewed or until automated QA passes.
- Sanitization for uploads: `DocumentIngestService` already blocks PDF actions and scripts; extend to detect embedded macros/archives and disallow attachments.
- Embedding integrity: embeddings must be generated by trusted service in-system; reject client-provided embedding vectors.
- Pgvector access: only allow similarity queries through parameterized helpers; disallow ad-hoc SQL strings from LLMs.

4. Data Layer / DB
- No dynamic SQL from external input: all raw queries must be parameterized and code-reviewed; add automatic code scan for `$executeRaw` / `$queryRaw` uses.
- Whitelist sortable/filter fields and map user inputs to column names explicitly.
- Audit DB changes (who, sessionId, input hash) and keep backups of deleted items for forensic review.

5. CI / PR checks
- Security linter (scripts/check_security_patterns.sh) runs at CI pre-merge and fails on:
  - use of `$executeRaw`/`$queryRaw` without an approved helper
  - patterns `exec(`, `spawn(`, `child_process`, `eval(`
  - direct embedding of vector strings coming from request bodies
- Tests: add integration tests for HITL enforcement and for embedding dimension validation.

Operational Playbook
- Incident detection: alerts on repeated failed HITL confirmations, sudden spike in `deleteDocument` requests, or changes to code paths that touch raw SQL.
- Response: automatically quarantine new ingests; require security review to unquarantine.
- Monitoring: instrument metrics (LLM calls/sec, tool actions/sec, deletions/sec) and set thresholds to PagerDuty/Slack.

Appendix
- Quick wins applied: HITL on delete endpoint, AuditService added, embedding validation in `DocumentRepository`.
